────────────────────────────────────────────────────────────
СЛАЙД 2. АКТУАЛЬНОСТЬ
────────────────────────────────────────────────────────────

Актуальность темы обусловлена тем, что спрос на персонализированные туристические услуги постоянно растёт, а объём информации о турах, отелях и перелётах увеличивается.

Сегодня клиент турфирмы тратит много времени на самостоятельный поиск: он просматривает сайты туроператоров, агрегаторы и системы бронирования. Менеджер, в свою очередь, вынужден вручную обрабатывать однотипные запросы: уточнять направление, даты, бюджет, искать подходящий тур и формировать предложение.

Это приводит к задержкам в ответе, потере клиентов и высокой нагрузке на сотрудников, особенно вне рабочего времени офиса.

Применение больших языковых моделей позволяет принимать запрос пользователя в свободной форме на русском языке, автоматически извлекать из него параметры поездки и формировать готовый турпакет на основе актуального каталога турфирмы.

Разработанный модуль «AI Тур-Генератор» работает на сайте круглосуточно и снижает нагрузку на менеджеров на этапе первичного подбора тура.

────────────────────────────────────────────────────────────
СЛАЙД 3. ЦЕЛЬ И ЗАДАЧИ
────────────────────────────────────────────────────────────

Цель работы — разработать и внедрить систему генерации турпакета на основе методов искусственного интеллекта, интегрируемую в веб-сайт туроператора ООО «Бон Вояж».

Для достижения цели были поставлены и решены следующие задачи:

1. Провести анализ деятельности туроператора и существующего процесса подбора туров.
2. Выполнить сравнительный анализ программных аналогов и обосновать выбор технологий.
3. Спроектировать архитектуру модуля и структуру базы данных.
4. Реализовать метод сбора данных с сайта туроператора.
5. Интегрировать большую языковую модель для извлечения параметров запроса и генерации турпакета.
6. Разработать бэкенд на FastAPI и фронтенд на Vue.js с диалоговым интерфейсом.
7. Провести тестирование и выполнить экономическое обоснование внедрения.

────────────────────────────────────────────────────────────
СЛАЙД 4. СРАВНЕНИЕ АНАЛОГОВ
────────────────────────────────────────────────────────────

При анализе аналогов рассматривались сервисы Travelpayouts, Tutu.ru, Яндекс.Путешествия, CRM-системы для турагентств, а также готовые чат-боты.

Поисковые системы предоставляют большую базу туров и актуальные цены, но работают только через формы с жёсткими полями. Они не понимают запрос в свободной форме и не ведут диалог для уточнения параметров.

CRM-системы, такие как «1С: Турагентство», решают задачи внутреннего учёта и документооборота, но не обеспечивают автоматический подбор тура клиентом на сайте и не используют технологии искусственного интеллекта.

Готовые чат-боты требуют ежегодной подписки, не связаны с каталогом конкретной турфирмы и не позволяют использовать собственный парсер данных.

Ни один из рассмотренных аналогов не сочетает одновременно три ключевые функции, необходимые предприятию:
— обработку запроса на естественном языке;
— диалог для уточнения параметров;
— работу с каталогом именно ООО «Бон Вояж».

Именно поэтому было принято решение о разработке собственного модуля.

Результатом работы стал программный модуль «AI Тур-Генератор», предназначенный для интеграции в сайт «Планета 360».

────────────────────────────────────────────────────────────
СЛАЙД 6. СРЕДСТВА РАЗРАБОТКИ
────────────────────────────────────────────────────────────

Система построена по клиент-серверной архитектуре.

Серверная часть реализована на Python 3.11 с использованием фреймворка FastAPI и сервера Uvicorn. FastAPI выбран за асинхронность, встроенную документацию API и удобную интеграцию с внешними сервисами.

Клиентская часть выполнена на Vue.js 3 в виде одностраничного приложения, которое встраивается в существующий сайт турфирмы.

Для хранения данных используется СУБД MySQL 8.0.

В качестве языковых моделей применяется связка GigaChat и DeepSeek. GigaChat обеспечивает качественную работу с русским языком и обработку персональных данных в соответствии с требованиям законодательства РФ. DeepSeek используется как резервный провайдер — при сбое основной модели система автоматически переключается на вторую.

Каталог туров обновляется парсером на Python с библиотеками BeautifulSoup и requests, который каждые 12 часов собирает данные с сайта bon-voyage28.ru.

Для подбора перелётов подключены API Travelpayouts и Amadeus.

────────────────────────────────────────────────────────────
СЛАЙД 7. БАЗА ДАННЫХ
────────────────────────────────────────────────────────────

Структура базы данных MySQL включает семь таблиц.

Таблицы hotels и flights хранят каталог туров и авиапредложения. Они заполняются парсером и внешними API соответственно.

Таблица user_sessions хранит состояние диалога каждого пользователя в формате JSON.

Таблица user_queries фиксирует все сообщения пользователя и извлечённые из них параметры.

Таблица generated_tours содержит сформированные турпакеты.

Таблица agency_leads сохраняет заявки клиентов, которые хотят обратиться к менеджеру.

Таблица daily_stats агрегирует статистику работы модуля: число обращений, сгенерированных туров и среднее время ответа.

Связи между таблицами user_sessions, user_queries, generated_tours и agency_leads отражены на ER-диаграмме, представленной на слайде.

────────────────────────────────────────────────────────────
СЛАЙД 8. ИНТЕРФЕЙС (ФРОНТЕНД)
────────────────────────────────────────────────────────────

Пользовательский интерфейс реализован в файле frontend/index.html.

Основной элемент — чат-диалог. Клиент вводит запрос свободным текстом, например: «Хочу в Китай с 10 по 20 июля, бюджет 80 000 рублей». Сообщения пользователя и ответы ассистента визуально разделены.

Пользователь может выбрать режим работы нейросети: автоматический, только GigaChat или только DeepSeek.

После сбора всех параметров система отображает карточку тура с названием отеля, датами, стоимостью перелёта, итоговой ценой и персональной рекомендацией.

Предусмотрены кнопки «Купить самостоятельно» — переход на страницу тура на сайте турфирмы, и «Обратиться в турфирму» — контактные данные менеджера.

Также есть форма заявки с полями имя, телефон и email, которая передаёт обращение менеджеру.

────────────────────────────────────────────────────────────
СЛАЙД 9. СЕРВЕРНАЯ ЧАСТЬ (БЭКЕНД)
────────────────────────────────────────────────────────────

Центральным компонентом системы является серверная часть на FastAPI, реализованная в модуле backend/app.py.

Основной эндпоинт — POST /api/chat. Алгоритм его работы следующий.

Первый шаг — создание или загрузка сессии пользователя. Состояние диалога хранится в оперативной памяти и дублируется в базе данных.

Второй шаг — обращение к языковой модели через модуль ai_service.py. Модель извлекает из текста направление, даты поездки и бюджет. Если данных недостаточно, ассистент задаёт один уточняющий вопрос.

Третий шаг — серверная валидация. Модули date_validation.py и dialog_hints.py проверяют корректность дат и дополняют параметры без участия нейросети.

Четвёртый шаг — поиск подходящих туров в таблице hotels по направлению и бюджету.

Пятый шаг — второй запрос к LLM для формирования финального турпакета с учётом найденных предложений и данных о перелёте.

При недоступности одного из провайдеров система автоматически переключается на резервный — GigaChat или DeepSeek.

Дополнительно реализованы эндпоинты сброса диалога, отправки заявки менеджеру и просмотра статистики.

ВОЗМОЖНЫЕ ВОПРОСЫ КОМИССИИ (краткие ответы)
────────────────────────────────────────────────────────────

В: Почему две нейросети?
О: Для отказоустойчивости. Если один провайдер недоступен, система переключается на второй без остановки сервиса.

В: Почему парсинг, а не API?
О: Сайт турфирмы не предоставляет открытый API к каталогу. Парсер обновляет данные каждые 12 часов автоматически.

В: Заменяет ли модуль менеджера полностью?
О: Нет. Модуль автоматизирует первичный подбор. Бронирование, оплата и юридическое оформление остаются за менеджером.

В: Как обеспечивается безопасность данных?
О: Секретные ключи хранятся в файле .env, не в коде. GigaChat обрабатывает данные в соответствии с 152-ФЗ.

В: Сколько таблиц в базе данных?
О: Семь: hotels, flights, user_sessions, user_queries, generated_tours, agency_leads, daily_stats.
